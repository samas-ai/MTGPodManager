## 📂 Architectural Decisions
*(Log specific choices made during the build here so future agents respect them)*
- 2026-06-10 — **Phase 7 Pass 1: design restyle + a11y (token-based, no behavior change).** "Arcane parchment" theme in `globals.css` (AA-contrast palette + `--mtg-w/u/b/r/g` mana tokens + `prefers-reduced-motion`); fonts via `next/font/google` (Cinzel display + Inter body) wired as `--font-display`/`--font-sans` in `layout.tsx` (also skip-to-content link + `#main`). New primitives: `ui/badge`, `ui/skeleton`, `ui/page-header`, `features/color-pips` (WUBRG pips pair color+letter, aria-labelled). Route states: `loading.tsx` (groups, groups/[id], stats/[id], profile/decks) + `error.tsx`/`not-found.tsx`/`global-error.tsx`. `CardTitle`/headings use display font; badges for admin/verified/imported; life-counter buttons aria-labelled; per-page `metadata.title`. Tokens-only (no raw hex in components) so a dark-mode toggle later is just a token swap.
- 2026-06-09 — **Supabase BaaS with RLS as the authorization boundary** (not a separate API service). Keep secrets/import/finalize logic strictly server-side; RLS is the net under that discipline.
- 2026-06-09 — **Match finalization lives in a `SECURITY DEFINER` Postgres function** `finalize_match(match_id, winner_id)`, not in app-layer code. An RLS `with check` forbids plain updates from flipping `status` to `finalized`; the function is the only door. Atomic, row-locked.
- 2026-06-09 — **Result model: winner-only** (`matches.winner_user_id`, validated to be a participant). Full finishing order deferred (clean future migration: add `placement` to `match_participants`).
- 2026-06-09 — **Live join-status transport: Realtime Postgres Changes** on `match_participants` filtered by `match_id`, with a **polling fallback already written** to degrade gracefully if free-tier Realtime limits bite. The DB is the single source of truth for participation.
- 2026-06-09 — **Life totals are local-only** (host screen React state). Synced life across devices is explicitly out of scope; keeps Realtime payloads tiny.
- 2026-06-09 — **Archidekt import is best-effort, server-side only**, with a manual two-field fallback (name + Scryfall-resolved commander). Deck *identity* (name + commander + color identity) is decoupled from full card list; `card_data` is nullable. No MVP stat depends on the card list.
- 2026-06-09 — **Snapshot deck identity onto `match_participants`** (`deck_name_snapshot`, `commander_snapshot`) so history stays correct after renames/deletes and `decks` can stay strictly owner-private.
- 2026-06-09 — **Stats via `security_invoker = on` views** (`group_player_winrates`, `group_deck_play_counts`) — group-scoped automatically via base-table RLS; recompute on read (fine at pod scale).
- 2026-06-09 — **Stack pinned (Phase 1 scaffold):** Next.js **14.2.35** (App Router, patched), React 18, `@supabase/ssr` **0.12** + `@supabase/supabase-js` **2.108** (must stay version-aligned — see quirk), Tailwind 3 + design-token CSS vars, shadcn-style UI primitives (copied-in, no Radix), Zod, Vitest + Playwright. No global state lib.
- 2026-06-09 — **Auth: email + password** (Supabase Auth). A `handle_new_user` trigger auto-creates the `profiles` row from sign-up metadata (`display_name`, falling back to email local-part).
- 2026-06-09 — **F1 membership RPCs** `create_group(p_name)`, `create_invite(p_group_id)`, `accept_invite(p_code)` are all `SECURITY DEFINER`; `group_members` has **no insert policy**, so RPCs are the only write path. Invite codes are 8-char uppercased slices of `gen_random_uuid()` (no pgcrypto dependency), 14-day expiry.
- 2026-06-09 — **DB types hand-authored** in `src/lib/supabase/database.types.ts` (no CLI/Docker here). Must mirror the generator's shape (`__InternalSupabase`, `Json`, `Enums`, `CompositeTypes`, `{ [_ in never]: never }` for empty Views/CompositeTypes). Regenerate with `supabase gen types typescript` once the CLI is available.
- 2026-06-10 — **F6 core group stats (final P0).** `0005_stats.sql` adds two `security_invoker = on` views — `group_player_winrates` (games/wins/win_rate per player) and `group_deck_play_counts` (most-played decks, finalized only) — group-scoped automatically via base-table RLS; `grant select to authenticated`. Reads via `lib/stats.ts` (pure `winPercent` + `getStandings`/`getDeckPlayCounts`/`getRecentMatches` taking the server client; win% computed in JS from wins/games). Full `/stats/[groupId]` page + compact teasers on pod home. pgTAP `stats_views.test.sql` confirms correct counts after finalize + non-member sees 0 rows. **MVP F1–F6 complete.**
- 2026-06-10 — **F5 join-to-verify & finalization (core differentiator).** `0004_finalize.sql` adds `finalize_match(p_match_id, p_winner)` SECURITY DEFINER — the ONLY path that sets `status='finalized'` (plain UPDATE blocked by `matches_update_open`). Row-locks the match; re-checks host, open, ≥2 participants, all verified+deck, winner-is-participant, atomically. `verifyParticipation` action upserts the caller's own `match_participants` row (RLS `mp_insert_self`/`mp_update_self`), snapshotting deck name/commander (history survives deck rename/delete; keeps `decks` owner-private). `finalizeMatch` action calls the RPC, maps exceptions to friendly copy, redirects to pod home. UI: join page `/match/[id]/join`, reusable `verify-deck.tsx` (used on join + host screens), finalize panel in `host-match.tsx` (winner select from live verified list), active-match discovery on group page. pgTAP `finalize_match.test.sql` covers all rejections + valid finalize + double-finalize + duplicate (9 assertions) — the suite that protects the trust guarantee.
- 2026-06-10 — **F4 host live session.** `0003_matches.sql` adds `matches` + `match_participants` + `match_status` enum + RLS (`matches_update_open` `with check status='open'` **blocks flipping to finalized** — finalize is F5's RPC) and publishes `match_participants` to `supabase_realtime`. **Verified:** Postgres Changes adheres to RLS per-subscriber (member-only `mp_select` scopes join events); DELETE events aren't RLS-filtered (we use INSERT/UPDATE only); auth-check is a high-throughput bottleneck (fine at pod scale). Life counter is **local-only** (`lib/match/life.ts` pure helpers; client `host-match.tsx`), 2–4 seats @ 40. Join status via `useParticipants` hook: Postgres Changes default, **polling fallback** via `NEXT_PUBLIC_REALTIME_TRANSPORT=polling` (re-queries every 3s). Host page is host-only (non-host → group; join page is F5). `startMatch` action inserts an open match (RLS `matches_insert_host`).
- 2026-06-09 — **F3 import: live Scryfall lookup + DB cache; identity + full card list.** `lib/services/scryfall.ts` (POST `/cards/collection`, ≤75 ids, `User-Agent`+`Accept`, 8s timeout), `lib/services/archidekt.ts` (best-effort GET `api/decks/{id}/`; pure `parseArchidektDeck` detects commander via the `isPremier` category, fixture-tested), `lib/services/import.ts` (orchestrator → `decks` insert with `source='archidekt'`, color-identity union, `card_data` = {commanders, cards}). Server-only modules (no `"use server"`); `importDeck` action lives in `services/decks.ts`. **No migration** — reuses F2 columns. Verified facts: Scryfall needs UA+Accept, `/cards/collection` max 75; Archidekt API is unofficial/undocumented/open-beta → best-effort + manual fallback is the guarantee. **No bulk-data download** (impractical on serverless; commander-only volume is tiny). Shared `Result<T>` in `src/lib/result.ts`.
- 2026-06-09 — **F2 decks: manual-only entry, strictly owner-private.** `0002_decks.sql` adds `decks` + `deck_source` enum + `decks_all_own` RLS (`user_id = auth.uid()` on read+write) + `touch_updated_at` trigger. F2 sets `source='manual'`, leaves `commander_scryfall_id` null / `color_identity` `{}` / `card_data` null — F3 import fills those by *updating* existing rows. Group-visible deck info will come from match-participant snapshots (F5), never from reading others' `decks`. Profile rename via `updateDisplayName` (uses existing `profiles_update_self`).

## 🐛 Known Issues & Quirks
*(Log current bugs or weird workarounds here)*
- 2026-06-10 — **`@vercel/analytics`/`@vercel/speed-insights` npm packages are uninstallable here:** their optional `@sveltejs/kit` peer chain demands vite@6, conflicting with Vitest 2's vite@5 pin (npm ERESOLVE even on v1.x). Solution shipped: first-party `/_vercel/...` scripts via `next/script` in `src/components/features/vercel-metrics.tsx` (zero deps, CSP `'self'` covers them). Revisit packages only if Vitest moves to vite@6+.
- 2026-06-10 — **CSP is production-only** (`NODE_ENV === "production"` in `next.config.mjs`) because `next dev` needs eval + localhost HMR websocket. If Scryfall card art ships (roadmap D1), add `https://cards.scryfall.io` to `img-src`.
- 2026-06-10 — **pgTAP fixture pattern (learned from 3 CI iterations):** suites that define `tests.act_as()` need `create schema if not exists tests; grant usage on schema tests to authenticated;` AND `grant select on <temp_table> to authenticated;` for any temp fixture table read while acting as a user (temp tables are owned by postgres). New suites must copy this preamble. Also: `config.toml` auth key is `[auth.email].enable_confirmations` in current CLI (bare `[auth]` placement fails `supabase start`).
- 2026-06-10 — **GitHub Actions Node 20 deprecation (deadline 2026-06-16):** CI log warns `actions/checkout@v4`, `actions/setup-node@v4`, `supabase/setup-cli@v1` run on Node 20, which gets force-switched to Node 24 on June 16 and removed Sept 16. Dependabot (github-actions ecosystem, weekly) should PR the bumps; if not, `ci.yml` needs manual version bumps — **protected area, needs user approval**.
- 2026-06-10 — **Editing repo files with PowerShell `Set-Content` corrupts UTF-8** (PS 5.1 reads BOM-less UTF-8 as ANSI → mojibake on em-dashes). Use the Edit tool or `sed` (byte-safe) for source edits.
- 2026-06-10 — **FIXED (migration 0008):** in production, creating a pod / saving a deck failed with FK 23503 "Key (created_by/user_id)=… not present in table profiles" — the signed-in user had no `profiles` row. Cause: the account was created before the `on_auth_user_created` trigger existed in that project (or the trigger wasn't present). `0008_backfill_profiles.sql` backfills profiles for all existing `auth.users` and re-asserts the function+trigger. **User must apply 0008.** Diagnostic: profile auto-creation depends on that trigger — verify it exists in any new Supabase project (Database → Triggers → `on_auth_user_created` on `auth.users`).
- 2026-06-10 — **FIXED (migration 0006):** deleting a deck that had been used in a match failed (FK violation 23503) because `match_participants.deck_id` was `ON DELETE NO ACTION`. Changed to **`ON DELETE SET NULL`** — the deck reference clears on delete while `deck_name_snapshot`/`commander_snapshot` preserve stats + history. **User must apply `0006_deck_delete_set_null.sql`** for the fix to take effect. `deleteDeck` now maps code 23503 to a clear message.
- ⚠️ **Verify before relying on it:** Supabase Realtime's RLS authorization behavior for Postgres Changes has changed across releases. Confirm current behavior in the Supabase docs during F4 setup. Design assumes a member-only `SELECT` policy on `match_participants` is sufficient to scope change events.
- ⚠️ **Re-check Archidekt ToS / endpoints before building F3.** No clearly documented, stable public API. If only an internal JSON path exists, treat as best-effort, low-frequency, server-side, cached — and lean on the manual fallback as the real guarantee.
- ⚠️ **Scryfall etiquette:** use bulk data + cache + batch `collection` endpoint; descriptive `User-Agent`; never per-card hammering.
- ⚠️ **Keep `@supabase/ssr` and `@supabase/supabase-js` version-aligned.** Mismatch (e.g. ssr 0.5 with supabase-js 2.108) scrambles the client's generic parameter order and silently resolves all query rows to `never` / RPC args to `undefined` — looks like a type bug but is a version skew. ssr 0.12 peer-deps supabase-js ^2.108. Bump both together.
- ⚠️ **No Docker/Supabase CLI in the dev environment** — pgTAP and `supabase start` can't run here. RLS is verified manually + in CI/Docker. Migrations are applied by hand via the dashboard SQL editor (see `SETUP.md`).
